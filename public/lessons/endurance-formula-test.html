The Endurance Formula
Understand the decay — then predict what your bot will do
Power Decay Formula
P(t) = 100 − (20 − E) × 1.5 × t
E = your Endurance stat  ·  t = time in minutes  ·  P(t) = motor % at that moment
Key Terms
P(t)Motor power remaining — starts at 100%, drops over time
Endurance (E)Your stat allocation (1–20). Higher E = slower decay = more consistent bot
decay rate20 − E. How fast your power drains each minute
tTime in minutes (0 to 3 for a full match)

⚠️ Two Problems — Before You Can Code This

The formula is mathematically correct. But MakeCode can't run it as written. Two things need to change.

Problem 1 — The Decimal (1.5)

MakeCode silently drops all decimals. When you type 1.5, the micro:bit reads 1. No error. No warning. Your decay rate gets cut in half and you'll never know why.

Fix: Replace 1.5 with 3 ÷ 2. Same math — but now MakeCode multiplies by 3 first, then divides by 2. No decimals anywhere.
Problem 2 — Time in Minutes

30 seconds is 0.5 minutes. MakeCode reads 0.5 as 0. So t = 0 every single loop. Zero times anything is zero — no decay ever happens. Your bot runs at full power the whole time.

Fix: Measure time in seconds. MakeCode gives you time in milliseconds — divide by 1000 to get whole seconds.
Transforming the Formula

Same prediction. New form that the micro:bit can actually calculate.

1
Original
P(t) = 100 − (20 − E) × 1.5 × t
Correct math — broken for the computer (decimal, time in minutes)
2
Replace 1.5 with 3÷2
P(t) = 100 − (20 − E) × 3 × t ÷ 2
Mathematically identical — no decimal anywhere
3
Replace t (minutes) with t÷60 (seconds)
P(t) = 100 − (20 − E) × 3 × (t ÷ 60) ÷ 2
Now t is in seconds — a whole number the computer can use
4
Collapse the division: 3 ÷ 2 ÷ 60 = 1 ÷ 40
P(t) = 100 − (20 − E) × t ÷ 40
Final form. Prediction unchanged — just a form the computer can run.

Program & Test
Build the decay code in MakeCode — then run your bot
Read this before you start.
Your bot will slow down during the test. The formula predicts one distance. The bot will probably do something slightly different. That gap is not a mistake — it's the whole point. You're comparing a math model to a real machine. Real machines have friction, battery differences, and floor variation. The gap is information.
The Code Structure

Your program has three parts: on start sets variables, on button A runs the decay loop, and on button A+B is the emergency stop.

on start:
  set SPEED to 0    // ← change to your stat
  set ENDURANCE to 0    // ← change to your stat
  set motorPower to SPEED × 100 ÷ 20
  set DECAY_RATE to 0    // ← you calculate: 20 − ENDURANCE
  set FLOOR to 20
  set DURATION to 0    // ← change to 180
on button A pressed:
  if not matchRunning then:
    set t to 0
    set matchStartTime to running time (ms)
    set matchRunning to true
    show string "GO"  →  clear screen
    while matchRunning:
      set t to (running time (ms) − matchStartTime) ÷ 1000
      if t ≥ DURATION then:
        set matchRunning to false  ·  stop all wheels  ·  break
      set currentPower to 100 − (DECAY_RATE × t ÷ 40)
      set currentPower to max of currentPower and FLOOR
      set left/right wheel speed motorPower × currentPower ÷ 100 %
      plot bar graph of currentPower up to 100
      pause 200 ms
on button A+B pressed:  // emergency stop
  set matchRunning to false  ·  stop all wheels  ·  show leds (✕)
The LED bar graph shows power draining in real time. The max of currentPower and FLOOR block replaces the if-statement — it's a cleaner way to enforce the 20% floor. A+B is the emergency stop at any time.

📏 How the Test Works

You don't need 2 meters of space. The bot only travels for 30 seconds at a time — you pick it up, return it to start, and let it run again. You are measuring how far the bot travels in the same 30-second window at three different moments in the match.

0:00 – 0:30
Run 1. Mark where the bot stops at 0:30. Pick it up.
1:30 – 2:00
Place bot at start at 1:30. Mark at 2:00. Pick it up.
2:30 – 3:00
Place bot at start at 2:30. Mark at 3:00. Measure all three.
📏 How to Measure
  1. Put a piece of tape on the floor at the starting line.
  2. After each 30-second run, mark where the front of the bot stopped.
  3. Measure from the starting tape to each mark using a tape measure.
  4. Record all distances in centimeters (cm).
▶ Run the Test
  1. 15
    Place the bot at the starting tape. Press A to start. Watch the LED bar graph — that's your power draining in real time.
  2. 16
    At 0:30 — mark the bot's position. Pick up the bot. The timer is still running.
  3. 17
    Before 1:30 — have the bot back at the starting line. At 1:30, set it down. Do not press A or B again.
  4. 18
    At 2:00 — mark the bot's position. Pick it up and place it back at the start by 2:30.
  5. 19
    At 2:30 — set the bot down for the final 30 seconds.
  6. 20
    At 3:00 — the bot stops on its own. Measure all three distances from the starting tape. Record in centimeters.
If the bot doesn't visibly slow down: Run the full 3 minutes. A bot with high Endurance (E = 15+) has a small decay rate — the difference is real but subtle. The numbers will tell the story.

Watch Out — Common Errors
Forgot ÷ 1000 on time
t is in milliseconds instead of seconds. Decay happens instantly and the bot dies in the first second.
While loop set to 30000 instead of 180000
Bot only runs 30 seconds. Match length is 3 minutes = 180,000 ms.
DECAY_RATE × (t ÷ 40) instead of DECAY_RATE × t ÷ 40
When t < 40, dividing t first gives 0. Always multiply first, divide last.
Missing floor check
Without set currentPower to max of currentPower and FLOOR, the bot may stop mid-run instead of running at minimum speed.
