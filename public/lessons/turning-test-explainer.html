Turning Test Program
A 20-second test program for the BBC micro:bit + Cutebot Pro. Flash it, press A, and watch your stat allocations turn into real robot motion — arcs, border recoveries, and endurance decay — before committing to a match build.
What each button does
Button A
Run 20-second test
Bot drives immediately using your current SPEED, ENDURANCE, and TURNING values. Runs until time expires, then stops and shows "DONE".
Button B
Cycle simulated time
Steps through 0 → 60 → 120 → 180 seconds of fake match time, so you can see how endurance decay changes motor power before the bot even moves.
A + B
Emergency stop
Kills all motors immediately and shows an X on the display. Use this if the bot heads somewhere it shouldn't. Takes effect within one check cycle (~60 ms).

How the arc pattern works

The drive loop alternates direction each cycle. Each cycle has Turning segments. Within a segment, one motor runs faster than the other by the turn ratio — creating an arc. Higher Turning = more arcs per cycle = tighter curves.

Arc preview — adjust sliders above to update
Motor assignment per segment
// lastTurnWasLeft alternates each cycle if (TURNING == 1) { // straight — both same power L = R = currentPower } else if (lastTurnWasLeft) { // arc right L = currentPower × (1 − turnRatio) R = currentPower × (1 + turnRatio) } else { // arc left L = currentPower × (1 + turnRatio) R = currentPower × (1 − turnRatio) }
Turning = 1: Bot drives straight. Turn ratio is still computed but both motors get identical power.
Turn ratio caps at 0.4. Turning values above 9 all produce the same ratio — the gap between motors is already at maximum.
Border detection and recovery

The bot checks sensors every 60 ms while driving. If any sensor reads above the threshold, it stops and runs a recovery sequence based on which side triggered.

Sensors 2 and 3 both map to return value 3 — treated as a head-on or corner collision. The response is a longer backup plus a wider pivot, then a second border recheck.

Pivot away
Left → pivot right. Right → pivot left. ±80 ms random jitter to avoid looping.

After pivoting, the code runs checkBorder() again. If the bot is still on the border, it backs up once more and pivots the opposite direction before resuming the drive loop.

The formulas

Every calculation in computeStats() maps directly to a stat from your allocation sheet. These are the same formulas used in the match engine — the test program shows you what they produce in real motion.

Base motor power
Speed × 100 ÷ 20
= Speed × 5
Same as the Speed stat lesson formula. Speed = 4 → 20% motor power.
Turn ratio
0.2 + Turning × 0.02
max 0.4
The percentage differential between left and right motors. Turning = 2 → ratio 0.24.
Arc segment duration
21 600 ÷ Turning
Total arc time (BASE_SEGMENT_MS × 12 = 21 600 ms) split evenly across arcs. Turning = 2 → 10 800 ms per arc.
Endurance remaining %
100 − (20 − E) × t ÷ 40
floor: 20%
Decay rate = 20 − Endurance. Multiplied by simulated time ÷ 40. Minimum 20% always.
Applied motor power (after decay)
round(basePower × enduranceFactor)
floor: 15%
Applied power at full match length can be significantly lower. This is what the motors actually receive. The 15% floor prevents the bot from being completely immobile.
Constants

These values are set at the top of the file. Changing them changes how aggressive the border recovery is and how often the bot checks for borders.

Constant Default What it controls
TEST_DURATION_MS 20 000 How long the bot runs when you press A. 20 seconds by default. Increase for a longer test.
BASE_SEGMENT_MS 1 800 Base unit for arc timing. Total arc time = BASE_SEGMENT_MS × 12. Raise this to make arcs longer.
BORDER_CHECK_MS 60 How often sensors are polled while driving (every 60 ms). Lower = faster reaction; higher = fewer checks per second.
BORDER_THRESHOLD 200 Grayscale reading above which a sensor counts as "on the border." Adjust if sensor readings don't match your mat.
BACKUP_MS 600 How long the bot reverses during border recovery. Longer = more clearance from the edge.
BORDER_PIVOT_MS 400 How long the pivot turn lasts after backing up. Longer = wider re-entry angle. Has ±80 ms random jitter.
BORDER_PIVOT_POWER 50 Fixed power used for all backup and pivot moves. Independent of Speed allocation.
How to use this program
Step-by-step
  1. Set SPEED, ENDURANCE, TURNING at the bottom of the file. Total must be ≤ 20.
  2. Flash to the micro:bit. Place bot in center of mat.
  3. Press B to set a simulated match time if you want to see decay. Otherwise stay at 0s.
  4. Press A. Bot drives for 20 seconds.
  5. Change Turning, reflash, and compare arc patterns side-by-side.
Changing Turning values: reflash the bot each time. The drive loop reads TURNING at runtime — no need to change any other code.
What to look for: does the arc fit inside the mat? Does the bot recover cleanly from corners? Does motor power drop visibly at 120 s vs 0 s?
If the bot gets stuck in a corner loop: press A+B to emergency stop. Increase BORDER_PIVOT_MS or BACKUP_MS and reflash.
